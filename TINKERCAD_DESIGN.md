# Arduino Visualization Design Document
### Visualization Tier — Tinkercad (Arduino UNO)

---

## 1. Subsystem Philosophy

In the simulation-first digital twin framework, the Tinkercad Arduino UNO acts strictly as an **I/O expander and visualization display driver**. 

To align with hardware boundaries, the Arduino **does not compute Dijkstra routes, calculate dynamic ETAs, or maintain independent traffic signal FSM logic**. It acts as a passive receiver, reading ASCII serial data packets generated by the FPGA control layer and directly mapping them to physical pins and character displays.

```
       [Virtual UART Channel] (mosi_serial_tx / uart_bus.txt)
                 │
                 ▼
    ┌──────────────────────────┐
    │   Hardware Serial RX     │ ◄─── Read incoming ASCII stream at 9600 Baud
    └────────────┬─────────────┘
                 │ (Character Stream)
                 ▼
    ┌──────────────────────────┐
    │   ASCII Protocol Parser  │ ◄─── Detect '$', extract states, mode, pos, ETA
    └────────────┬─────────────┘
                 │ (Decoded State Enums)
                 ▼
         ┌───────┴───────┐
         ▼               ▼
    ┌──────────┐   ┌───────────┐
    │ LED Pins │   │  16x2 LCD │ ◄─── Print routing updates and mode strings
    └──────────┘   └───────────┘
```

## 2. Hardware Pin Mapping

The Tinkercad virtual breadboard connects the Arduino UNO pins to the physical LEDs, buttons, and character display module:

### A. Traffic Signal LEDs (Pins D2 – D10)
*   **Junction A:** Red = `D2`, Yellow = `D3`, Green = `D4`
*   **Junction B:** Red = `D5`, Yellow = `D6`, Green = `D7`
*   **Junction C:** Red = `D8`, Yellow = `D9`, Green = `D10`

### B. Interactive Inputs (Pins D11 – A0)
*   **Junction A Vehicle Counter Button:** Pin `D11`
*   **Junction B Vehicle Counter Button:** Pin `D12`
*   **Junction C Vehicle Counter Button:** Pin `D13`
*   **Ambulance Emergency Trigger Button:** Pin `A0` (configured with pull-down resistor)

### C. LiquidCrystal Display (Pins A1 – D1)
*   **Register Select (RS):** Pin `A1`
*   **Enable (EN):** Pin `A2`
*   **Data Pins (D4, D5, D6, D7):** Pins `A3`, `A4`, `A5`, `D1`

## 3. Serial Communication Parser

The Arduino sketch runs a polling parsing routine on the UART serial interface (`Serial.read()`).

### A. Protocol String Syntax
`$` `[SigA]` `[SigB]` `[SigC]` `[Mode]` `[Pos]` `[ETA_1]` `[ETA_0]` `\n`
*   `$`: Byte 0 (Start Byte)
*   `SigA`, `SigB`, `SigC`: Bytes 1–3 (Signal state code: 0 = RED, 1 = YELLOW, 2 = GREEN, 3 = EMG_GREEN)
*   `Mode`: Byte 4 (System mode: 0 = Normal, 1 = Route, 2 = Corridor, 3 = Arrival, 4 = Recovery)
*   `Pos`: Byte 5 (Ambulance position junction letter)
*   `ETA_1`, `ETA_0`: Bytes 6–7 (Two-digit remaining ETA minutes)
*   `\n`: Byte 8 (Newline End Byte)

### B. Display and LED Write Logic
*   **LED Control:** If `SigA == 0`, `digitalWrite(RED_A, HIGH)`, else `LOW`. If `SigA == 1`, `digitalWrite(YEL_A, HIGH)`, else `LOW`. If `SigA == 2` or `3`, `digitalWrite(GRN_A, HIGH)`, else `LOW` (where `3` triggers flashing on green if hardware indicates dynamic clearing).
*   **LCD Control:**
    *   If `Mode == 0` (NORMAL): Print junction traffic densities (`A:LOW B:MED C:LOW`).
    *   If `Mode == 1` (ROUTE): Print path routing (`ROUTE: A>D>G>H>I`).
    *   If `Mode == 2` (CORRIDOR): Print position and ETA (`AMB@D ETA: 02:40`).
    *   If `Mode == 3` (ARRIVAL): Print destination success (`AMB AT HOSPITAL / SAVED: 06:00`).
    *   If `Mode == 4` (RECOVERY): Print system reset warning (`RECOVERING...`).

## 4. Verification and Simulation Steps

1.  **Tinkercad Serial Feeder:** Open the Tinkercad circuit dashboard, activate the Serial Monitor window, and input the ASCII packet string (e.g., `$2002D02\n`).
2.  **Visual Output Check:** Verify that:
    *   Junction A Green LED lights up.
    *   Junction B & C Red LEDs light up.
    *   LCD screen prints: `CORRIDOR ACTIVE / AMB@D  ETA:02:00`.
3.  **Sensor Input Verification:** Pressing `D11` increases the count on the LCD in `NORMAL` mode. Pressing `A0` immediately prints the route calculations string.
