# Hardware Migration Plan
## Simulation-to-Silicon (Digital Twin to Physical Hardware) Transition Guide

---

## 1. Overview

This project is built using a **simulation-first digital twin** architecture. Every embedded interface, register, and state machine is designed to represent real physical hardware. 

This migration plan outlines how the virtualized interfaces and communication layers (Virtual SPI, Virtual UART) can be replaced with physical hardware connections between a **physical STM32 MCU**, a **physical FPGA**, and a **physical Arduino/LCD/LED setup** with minimal modifications to the core business logic.

```
       SIMULATION-FIRST DIGITAL TWIN
       ┌───────┐         Virtual SPI         ┌────────┐         Virtual UART        ┌─────────┐
       │ STM32 │ ──────────────────────────▶ │ Vivado │ ──────────────────────────▶ │ Arduino │
       └───────┘         (Log Files)         └────────┘         (Log Files)         └─────────┘
           │                                      │                                      │
           ▼ (1-to-1 Mapping)                     ▼ (1-to-1 Mapping)                     ▼ (1-to-1 Mapping)
       ┌───────┐         Physical SPI        ┌────────┐         Physical UART       ┌─────────┐
       │ STM32 │ ──────────────────────────▶ │  FPGA  │ ──────────────────────────▶ │ Arduino │
       │ Board │       (Hardware Pins)       │ Board  │       (Hardware Pins)       │  Board  │
       └───────┘                             └────────┘                             └─────────┘
```

## 2. STM32 Subsystem Migration

In simulation, the STM32 writes the packed 32-bit SPI frame into a local data array and logs it to a text file. In hardware, this is replaced by the physical **SPI1 peripheral** of the STM32 (configured as Master).

### A. STM32 SPI Master Hardware Settings
*   **Mode:** 2-Line Uni-directional Master (Transmit Only)
*   **Direction:** Full Duplex / Simplex Transmit Only
*   **Data Size:** 8 Bits (or 16-bit packed word)
*   **Clock Polarity (CPOL):** Low (0)
*   **Clock Phase (CPHA):** 1 Edge (0) - Data is latched on the rising edge of SCLK
*   **Baud Rate Prescaler:** `SPI_BAUDRATEPRESCALER_16` (providing ~4 MHz SCLK on a 64 MHz MCU clock, well within standard FPGA input buffer setup times)
*   **NSS Management:** Software (controlled manually via a GPIO pin for Chip Select)

### B. Pin Mapping Constraints
On a typical STM32F103C8 (Blue Pill) board, the SPI1 master lines are mapped to the following GPIOs:

| Pin Name | STM32 Pin | FPGA Connection | Type | Description |
|---|---|---|---|---|
| **SPI1_SCK** | `PA5` | FPGA PMOD SCK | Output | Clock signal generated by Master. |
| **SPI1_MISO**| `PA6` | FPGA PMOD MISO| Input | Data return path from Slave. |
| **SPI1_MOSI**| `PA7` | FPGA PMOD MOSI| Output | Data transfer from Master to Slave. |
| **SPI1_NSS** | `PA4` | FPGA PMOD CS/SS| Output | GPIO output manually driven LOW before transmission. |

### C. STM32 Driver API Migration Code
To bridge the virtual output function to physical hardware, we modify the transmit driver in `spi.c` using the STM32 HAL library:

```diff
 void SPI_Transmit_Frame(SpiFrame* frame) {
-    // Virtual Logging (Simulation)
-    write_frame_to_log(frame->raw_data, "simulation/virtual_bus/spi_bus.txt");
-    printf("Virtual SPI Out: 0x%08X\n", frame->raw_data);
+    // Physical Hardware Transmission (HAL SPI)
+    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_RESET); // Select Slave (CS = LOW)
+    
+    uint8_t tx_buffer[4];
+    tx_buffer[0] = (frame->raw_data >> 24) & 0xFF;
+    tx_buffer[1] = (frame->raw_data >> 16) & 0xFF;
+    tx_buffer[2] = (frame->raw_data >> 8)  & 0xFF;
+    tx_buffer[3] = (frame->raw_data)       & 0xFF;
+    
+    HAL_SPI_Transmit(&hspi1, tx_buffer, 4, 100); // Send 32-bits (4 bytes)
+    
+    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_SET);   // Release Slave (CS = HIGH)
 }
```

## 3. FPGA Subsystem Migration

In simulation, Vivado testbenches force parallel register values. For physical deployment, we synthesize the **SPI Slave Receiver core** (`spi_slave_receiver.v`) and map its inputs to physical board pins using Xilinx Design Constraints (XDC).

### A. Pin Constraints File (`green_corridor.xdc`)
For a standard Artix-7 board (e.g., Basys 3 or Nexys A7), PMOD header pins are assigned for SPI and UART lines:

```xdc
## Clock Signal (100 MHz on-board oscillator)
set_property PACKAGE_PIN W5 [get_ports clk]							
	set_property IOSTANDARD LVCMOS33 [get_ports clk]
	create_clock -add -name sys_clk_pin -period 10.00 -waveform {0 5} [get_ports clk]

## Reset Button
set_property PACKAGE_PIN V17 [get_ports rst]						
	set_property IOSTANDARD LVCMOS33 [get_ports rst]

## SPI Slave Port (Mapped to PMOD Header JA)
set_property PACKAGE_PIN J1 [get_ports nss]							
	set_property IOSTANDARD LVCMOS33 [get_ports nss]
set_property PACKAGE_PIN L2 [get_ports mosi]							
	set_property IOSTANDARD LVCMOS33 [get_ports mosi]
set_property PACKAGE_PIN J2 [get_ports sclk]							
	set_property IOSTANDARD LVCMOS33 [get_ports sclk]
set_property PACKAGE_PIN G2 [get_ports miso]							
	set_property IOSTANDARD LVCMOS33 [get_ports miso]

## UART Serial Transmit Output (Mapped to PMOD Header JB)
set_property PACKAGE_PIN A14 [get_ports uart_tx]						
	set_property IOSTANDARD LVCMOS33 [get_ports uart_tx]
```

### B. Hardware Synchronization (CDC)
Because the SPI Master SCLK is asynchronous to the internal 100 MHz FPGA clock, double-stage registers must be mapped to prevent metastability on the received control flags:

```verilog
reg [31:0] synced_rx_data;
always @(posedge clk or posedge rst) begin
    if (rst) begin
        synced_rx_data <= 32'b0;
    end else if (data_ready_synchronized) begin
        synced_rx_data <= rx_data_parallel;
    end
end
```

### C. UART Logger to Physical Baud Rate Generator
The virtual logger is replaced with a physical UART Transmitter module configured to output at 9600 Baud:
$$\text{Baud Rate Generator Divisor} = \frac{100\text{ MHz}}{9600 \times 16} = 651$$

## 4. Tinkercad to Arduino UNO Migration

In Tinkercad, the Arduino code calculates dummy routes and runs normal cycling. In the hardware digital twin model, the Arduino is demoted to a simple visual display controller, receiving UART instructions from the FPGA.

### A. Arduino Pin mapping
The physical LCD and LED signals are wired as follows:

```
        FPGA BOARD (Artix-7)                 ARDUINO UNO BOARD
        ┌──────────────────┐                 ┌────────────────┐
        │  UART_TX Pin     ├────────────────▶│  D0 (RX Pin)   │
        │  GND Pin         ├────────────────▶│  GND Pin       │
        └──────────────────┘                 └────────────────┘
```

*   **Junction A LEDs:** Red = `D2`, Yellow = `D3`, Green = `D4`
*   **Junction B LEDs:** Red = `D5`, Yellow = `D6`, Green = `D7`
*   **Junction C LEDs:** Red = `D8`, Yellow = `D9`, Green = `D10`
*   **Buttons:** Vehicle inputs = `D11`, `D12`, `D13`, Emergency = `A0`
*   **16x2 LCD display:** RS=`A1`, EN=`A2`, D4=`A3`, D5=`A4`, D6=`A5`, D7=`D1`

### B. Hardware Arduino Code (Unified Terminal)
The physical Arduino sketch replaces FSMs and route calculations with a serial frame parser:

```cpp
#include <LiquidCrystal.h>
LiquidCrystal lcd(A1, A2, A3, A4, A5, 1);

void setup() {
  Serial.begin(9600); // Hardware Baud Rate matching FPGA UART Output
  lcd.begin(16, 2);
  lcd.clear();
  lcd.print("HARDWARE ACTIVE");
}

void loop() {
  if (Serial.available() >= 9) {
    char start = Serial.read();
    if (start == '$') {
      int sigA = Serial.read() - '0';
      int sigB = Serial.read() - '0';
      int sigC = Serial.read() - '0';
      int mode = Serial.read() - '0';
      char pos  = Serial.read();
      int eta   = (Serial.read() - '0') * 10 + (Serial.read() - '0');
      
      updatePhysicalLEDs(sigA, sigB, sigC);
      updatePhysicalLCD(mode, pos, eta);
    }
  }
}
```

## 5. Bring-up & Verification Strategy

### Step 1: SPI Master loopback Test (STM32)
*   Short `PA7 (MOSI)` and `PA6 (MISO)` on the STM32 board.
*   Transmit a test frame and verify that the received SPI buffer exactly matches the sent array.

### Step 2: SPI Frame Logic Verification (FPGA)
*   Program the Artix-7 with the synthesized SPI Slave logic.
*   Use a logic analyzer or Vivado ILA (Integrated Logic Analyzer) IP to probe internal signals: `rx_data_parallel` and `data_ready`.
*   Ensure checksum errors flag when bits are intentionally corrupted using pin grounding.

### Step 3: Closed-Loop Integration Test
*   Connect the PA5, PA7, PA4 pins of STM32 to the PMOD ports of the FPGA.
*   Connect the UART TX line of the FPGA to the Arduino RX line.
*   Trigger an emergency route from PA0. Verify that the Tinkercad equivalent LCD displays the correct coordinates dynamically mapped by the STM32.
